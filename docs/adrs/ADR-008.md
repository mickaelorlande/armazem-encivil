# ADR-008 — Remover code-splitting por rota (bundle único)

**Data:** 2026-06-22
**Estado:** Aceite

---

## Contexto

`routes.tsx` usava `lazy(() => import('./pages/X'))` para cada página, gerando um chunk JS separado por rota com hash de conteúdo no nome (ex: `HistoryPage-CVsWeNl0.js`). Após cada deploy na Vercel, os hashes mudam. Utilizadores com a app já aberta (ou com o service worker da PWA a servir um `index.html` em transição) tentavam carregar um chunk antigo que já não existe na URL de produção — a Vercel só serve ficheiros do deployment mais recente. Resultado: erro `Failed to fetch dynamically imported module` ao navegar para Histórico, Relatórios, Configurações ou Ajuda, sempre nas horas seguintes a um deploy.

Duas tentativas de mitigação ficaram no código antes desta decisão:
1. `window.addEventListener('vite:preloadError', () => reload())` em `main.tsx`
2. `errorElement` customizado em `routes.tsx` com botão "Recarregar App"

Ambas reduziam o problema mas não o eliminavam — o service worker (`skipWaiting: true` + `clientsClaim: true`) por vezes servia o `index.html` antigo de novo no reload, criando um loop até à terceira tentativa.

## Decisão

Remover `lazy()` de todas as rotas. Todas as páginas são importadas diretamente em `routes.tsx`. O build gera um único bundle JS (~300 KB gzip) em vez de ~12 chunks separados.

## Consequências

**Positivas:**
- Stale-chunk torna-se estruturalmente impossível — não há chunks separados que possam "desaparecer" entre deploys
- Simplifica `routes.tsx` (sem `Suspense`/`PageLoader` por rota)
- Para uma PWA instalada por 3-5 utilizadores internos, o bundle é descarregado uma vez e fica em cache

**Negativas:**
- Primeiro load ~300ms mais lento em ligação 3G (aceitável dado o contexto de uso interno e PWA)
- Qualquer alteração a uma página invalida o cache do bundle inteiro (vs. só o chunk dessa página) — irrelevante à escala atual de tráfego

Mantivemos os dois mecanismos de recuperação (`vite:preloadError` + `errorElement`) como defesa em profundidade, agora também a limpar caches do service worker antes de recarregar, para qualquer cenário futuro de erro de import dinâmico (ex: se code-splitting for reintroduzido para uma feature específica).

## Alternativas Consideradas

- **`output.manualChunks` com nomes estáveis:** reduz a frequência de hashes novos mas não elimina o problema — só adia.
- **Cache-busting mais agressivo do service worker (`NetworkFirst` para HTML):** resolve parcialmente mas não remove a complexidade; ainda há uma janela de transição.
- **Bundle único (escolhida):** elimina a classe de problema por completo. Trade-off aceitável dado o tamanho real do app e o número de utilizadores.
