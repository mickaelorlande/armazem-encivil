# ADR-001 — Usar Supabase em vez de servidor local

**Data:** 2026-06-11  
**Estado:** Aceite

---

## Contexto

O projeto precisa de uma base de dados, autenticação e API. A ENCIVIL não tem infraestrutura de servidor para manter, e a equipa técnica é pequena.

## Decisão

Usar Supabase como backend gerido (BaaS — Backend as a Service), eliminando a necessidade de servidor dedicado.

## Consequências

**Positivas:**
- Zero gestão de infraestrutura
- Auth, DB, Storage e API incluídos num serviço
- RLS nativa no PostgreSQL — segurança sólida sem lógica de servidor
- SDK TypeScript com types gerados automaticamente
- Dashboard web para gestão direta da BD
- Backups automáticos

**Negativas:**
- Dependência de fornecedor (vendor lock-in)
- Limite de features no plano gratuito (500MB DB, 50.000 MAU)
- Latência pode variar (servidor em região europeia — Paris)

## Alternativas Consideradas

- **Node.js + Express + PostgreSQL próprio:** Maior controlo, mas requer DevOps e manutenção constante.
- **Firebase:** Boa alternativa, mas NoSQL dificulta relatórios e auditoria com SQL.
- **PocketBase:** Self-hosted, mais simples, mas sem a maturidade do Supabase.
